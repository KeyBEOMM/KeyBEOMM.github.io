# Stage 1-0: Kernel Architecture & Determinism (RTOS 심층 원리)

본 챕터에서는 로봇 제어를 위한 Real-Time Operating System(RTOS)의 커널 동작 원리와 설계 철학을 심층적으로 다룹니다.

---

## 1. Intro. RTOS를 이해하기 위한 필수 용어 4가지

### ① Preemptive Scheduling (선점형 스케줄링)
RTOS는 **우선순위(Priority)**의 절대 권력을 따릅니다.
우선순위가 가장 높은 응급 Task가 발생하는 순간, RTOS의 커널(스케줄러)은 현재 실행 중이던 낮은 우선순위의 Task가 뭐든 간에 그 작업을 **강제로 중단시키고(Preempt) CPU 컨트롤을 빼앗아** 응급 Task에게 넘겨줍니다.

### ② Timeslice (시분할)
만약 절대 권력을 가진 높은 우선순위의 Task가 2개(동일한 우선순위)가 있다면 어떡할까요?
RTOS는 시스템 틱(Tick, 보통 1ms 단위)마다 잘게 쪼개서 두 Task에게 공평하게 CPU 자원을 번갈아 할당합니다. 무언가가 혼자서 CPU를 독점하는 것을 시스템 차원에서 차단하고 잘게 나누는 것을 시분할이라고 합니다.

### ③ Context Switch (문맥 교환)
스케줄러가 A 작업을 멈추고 B 작업으로 방을 빼줄 때 딜레이 없이 짠 하고 바뀌지 않습니다.
A 작업이 하던 레지스터 값과 변수 상태를 어딘가(Stack)에 고이 접어 보관해두고, B 작업이 예전에 멈췄던 지점의 상태를 다시 불러와야 합니다. 이를 'Context Switch'라고 하며, 필연적으로 **오버헤드(시간 소모)**가 발생합니다.

### ④ Multitasking vs Super Loop (아두이노 `loop()`의 한계)
아두이노는 하나의 거대한 `while(1)`(Super Loop) 안에서 순서대로 코드가 돕니다.
```cpp
void loop() {
    readSensor(); // 1. 센서 읽기 (2ms)
    calculate();  // 2. 머리 굴리기 (5ms)
    moveMotor();  // 3. 모터 굴리기 (3ms)
}
```
만약 어디선가 센서가 고장 나서 `readSensor()`가 무한 대기(Block)에 빠지면 어떻게 될까요? 아래에 있는 모터 구동 코드도 전부 얼어붙어 로봇이 쓰러집니다.
반면 RTOS 기반 아키텍처는 센서, 통신, 모터 제어가 각각 **독립적인 Task 공간**에서 돌기 때문에, 특정 기능이 고장 나도 코어 제어 루프는 살아남아 모터를 비상 정지시킬 수 있습니다.

## 2. Real-Time 시스템의 명확한 정의 (Determinism & Classification)

Real-Time의 본질적인 전산학적 의미는 연산이 단순히 "빠르다(Low Latency)"가 아니라 지정된 **마감 시간(Deadline) 안에 논리적, 시간적 정확성을 100% 보장(Determinism)**하는 것입니다.

* **Hard Real-Time:** Deadline 초과 시 **System Failure** 및 치명적 물리 사고를 유발하는 시스템. (예: 로봇 다리의 관절 제어 루프, ABS 브레이크 MCU).
* **Firm Real-Time:** Deadline을 초과한 결과 데이터는 가치가 `0`이 폐기되어 서비스 품질은 떨어지지만 시스템 전체의 Failure로 직결되지는 않는 시스템.
* **Soft Real-Time:** Deadline을 부분적으로 초과해도 허용되며, 지연된 응답이 여전히 유효한 시스템. (예: 네트워크 패킷 파싱, 영상 스트리밍 버퍼).

> 로봇의 역운동학(Kinematics) 기반 동역학 제어 루프는 대표적인 **Hard Real-Time** 도메인입니다. 0.02초(50Hz) 주기에 맞춰 타겟 각도를 산출 및 갱신하지 못하고 스케줄러 디스패치 지연이 끼어드는 순간 궤적은 발산하고 시스템은 직립 밸런스를 잃고 붕괴됩니다.

## 3. GPOS vs RTOS Kernel Scheduling 원리 비교

우리가 매일 쓰는 Windows, MacOS, Linux, iOS, Android 등은 **GPOS (General Purpose Operating System, 범용 운영체제)** 라고 부릅니다.
반면 FreeRTOS, VxWorks, QNX 등은 **RTOS (Real-Time Operating System, 실시간 운영체제)** 라고 부릅니다.

일반 운영체제(Linux, Windows)와 RTOS(FreeRTOS 등)는 커널 스케줄러의 설계 철학이 완전히 상반됩니다.

### GPOS (General Purpose OS)

"최대한 많이, 공평하게 처리하자" (Throughput & Fairness)
스마트폰이나 PC는 사용자가 크롬 창을 10개 띄우고 유튜브를 보면서 카카오톡을 해도 버벅대지 않고 부드럽게 돌아가야 합니다.
GPOS는 전체 시스템의 **처리량(Throughput)**을 늘리는 것에 집중합니다. 만약 CPU가 바빠서 카카오톡 전송이 0.1초 늦게 갔다고 해서 세상이 무너지지 않습니다

* **목적 지향점:** 시스템 전체의 **Throughput(처리량 극대화)** 및 다중 사용자에 대한 **Fairness(공평성)**.
* **스케줄러 아키텍처 (예: Linux CFS - Completely Fair Scheduler):** 모든 프로세스에게 시스템 틱마다 런타임을 부여하며, CPU 독점을 막기 위해 Virtual Runtime이 누적된 프로세스를 우선순위 후순위로 강등(Aging)시킵니다.
* **임베디드 제어 한계:** 커널 내에 예측 불가능한 I/O 블로킹 타이밍이 존재하며, 특권 데몬들이 스케줄링 간섭을 일으킵니다. 최상위 우선순위를 부여하더라도 Dispatch Latency(인터럽트 발생부터 타겟 태스크 수행 전까지의 지연)를 수학적으로(Worst-Case Execution Time, WCET) 상한선(Upper bound) 지을 수 없습니다.

### RTOS (Real-Time OS)

"약속한 시간(Deadline)을 무조건 지킨다" (Determinism)

로봇, 자동차 ECU, 심박동기, 로켓 제어 시스템은 어떨까요?
100km/h로 달리는 자동차에서 브레이크를 밟았는데, OS가 구글 맵을 렌더링하느라 브레이크 연산을 0.1초 늦게 처리했다면? 곧바로 대형 사고입니다.
RTOS는 연산 속도 자체가 빠른 것이 아닙니다. 자신이 맡은 특정 연산을 **수행해야 할 정확한 시간에 지연 없이 칼같이 수행(정시성, Determinism)**하는 시스템입니다.

* **목적 지향점:** 극단적인 **Predictability(예측 가능성)** 중심.
* **스케줄러 아키텍처 (Strict Priority-based Preemptive Scheduling):** 엄격한 우선순위 기반 선점형 스케줄러입니다. 가장 높은 우선순위(Ready)의 Task가 깨어나면, 현재 실행 중인 Task가 무엇이든 커널 틱(SysTick)이 발생하거나 인터럽트가 끝나는 적시점에 **즉각적으로 하드웨어 레지스터 권한을 탈취(Preempt)**합니다. '공평함' 알고리즘은 배제됩니다.

## 4. Kernel Mechanics: Context Switch와 TCB (하드웨어 오버헤드 분석)

### ❓ 스레드(Thread/Task)란 무엇인가요?
스레드(RTOS에서는 주로 Task라고 부름)는 **'CPU가 실행하는 독립적인 작업의 단위'**입니다. 
비유하자면, 요리사(CPU 코어)가 1명뿐인 식당에서 '파스타 만들기(Task A)'와 '스테이크 굽기(Task B)'라는 두 개의 레시피를 동시에 진행해야 할 때, 각각의 레시피 처리 흐름을 독립적인 '스레드'라고 부릅니다. 
식당 코어는 1개이므로 요리사는 파스타를 젓다가 멈추고, 스테이크를 뒤집다가 멈추는 식으로 두 작업을 쉴 새 없이 왔다 갔다(동시성, Concurrency) 하며 마치 두 요리가 동시에 만들어지는 것처럼 보이게 합니다.

### ❓ 하드웨어 비용(Context Switching Overhead)이란 무엇인가요?
> "멀티 스레드가 칩 레벨에서 동시성(Concurrency)을 띄는 과정에서 발생하는 하드웨어 비용" 
이 문장은 **"요리사가 파스타를 만들다가 스테이크로 넘어갈 때, 손(CPU 레지스터)을 한 번 씻고 레시피(메모리)를 교체하느라 낭비되는 시간"**을 뜻합니다. 
CPU는 1개인데 여러 Task를 번갈아 처리하다 보니, A 작업을 멈추고 B 작업으로 넘어갈 때 딜레이(오버헤드)가 필연적으로 발생합니다. 이 교체 비용을 무시하고 스레드를 수십 개씩 무작정 만들면 CPU는 실제 일(제어 연산)은 안 하고 레시피 교체만 하다가 시간을 다 날리게 됩니다.

---

### Task Control Block (TCB) 쉽게 이해하기
TCB는 알기 쉽게 말해 **"스레드들의 신분증이자 작업 이력서 진도표"**입니다.
요리사(CPU)가 파스타(Task A)를 끓이다가 긴급하게 스테이크(Task B)로 넘어갈 때, *"파스타는 면을 3분까지 끓였고 소금은 2스푼 넣고 대기 중이었음"* 이라는 **진도표**를 어딘가에 적어둬야 나중에 파스타로 돌아왔을 때 정확히 이어서 요리할 수 있겠죠? 이 역할(작업 상태 보존)을 하는 메모리상 구조체가 바로 TCB입니다.

FreeRTOS는 `xTaskCreate` 호출 시 동적 메모리(Heap) 혹은 정적 영역에 메타데이터 객체인 TCB를 인스턴스화 합니다.

FreeRTOS는 `xTaskCreate`로 새로운 스레드를 만들 때, 메모리에 이 TCB 신분증을 발급해 줍니다.
즉, FreeRTOS는 'xTaskCreate'를 통해 HEAP 혹은 static 메모리 영역에 TCB를 생성한다.

주요 이력서 내용(구조체 멤버)은 다음과 같습니다.
* `pxTopOfStack`: 현재 Task가 어디까지 실행됐는지 적어놓은 메모리의 책갈피(Stack Pointer). 교체되어 쫓겨날 때 현재 CPU의 모든 상태를 묶어 이 주소 공간에 저장합니다.
* `uxPriority`: 이 Task의 계급장(우선순위). 우선순위 5인 Task와 1인 Task가 동시에 요리해달라고 요청하면, 커널 스케줄러는 이 TCB를 확인하고 5인 Task에게 CPU를 넘겨줍니다.
* `xStateListItem` / `xEventListItem`: 커널의 Ready, Blocked 리스트 등과 같은 이중 연결 리스트(Doubly Linked List)에 TCB 노드를 삽입/제거하기 위한 포인터.

### 하드웨어 레벨의 Context Switch (교체) 과정
높은 우선순위 Task로 권한이 넘어갈 때, CPU 코어 내부에서는 다음과 같은 쌩고생을 합니다:
1. **상태 백업 (Pushing State):** 파스타(Task A)를 멈춥니다.(매모리 파이프라인 지연) CPU 안에 있던 레지스터(R0~R15, 작업 중이던 임시 계산 데이터)를 A 전용 스택 영역애 최상단에(`PUSH`).
2. **이력서 갱신 (Context Update):** 커널은 A Task의 TCB의 stack pointer(`pxTopOfStack`)를 갱신 -> 현재 cpu의 모든 상태를 저장시켜놓는 것
3. **다음 타자 색출 (Queue Manipulation):** 커널이 대기줄(Ready List)을 순회(O(1)) 혹은 연산(O(n))하여 최상위 우선순위 TCB를 선택(B Task의 TCB)
4. **작업 복원 (Restoring State):** B Task의 TCB pointer를 통해 메모리를 찾아가서(참조하여), 과거 레지스터 값들을 CPU 코어로 꺼내와(`POP`) CPU를 이전 상태로 restore하고 program counter를 점프시킨다

 - 프로그램 카운터(Program Counter, PC)는 CPU 내부 레지스터 중 하나로, 다음에 실행할 명령어의 메모리 주소를 가리키는 포인터입니다. 명령어 포인터(Instruction Pointer, IP)라고도 부르며, 명령어가 실행될 때마다 자동으로 다음 주소로 증가하여 프로그램의 순차적 흐름을 제어하는 핵심적인 역할을 합니다. 

5. **파이프라인 초기화 (Pipeline Hazard):** 이 갑작스러운 교체 과정에서 CPU가 성능을 올리기 위해 미리 읽어뒀던 다음 명령어들(Instruction Cache)이 싹 다 쓸모 없어지고(cashe miss 발생!!) cpu pipeline의 Instruction buffer가 깨지게된다. 

 - 명령어 버퍼(Instruction Buffer)는 CPU가 메모리에서 미리 가져온 명령어를 임시로 저장하는 고속 메모리 영역입니다. 파이프라인 구조에서 명령어 인출(Fetch) 단계를 병목 현상 없이 가속화하여 CPU 성능을 향상시키고, 분기(Branch) 발생 시 오버헤드를 줄이는 역할을 합니다. 

> 📝 **요점:** RTOS는 코드를 논리적 단일 책무 단위별로 쪼개기 좋게 하지만, **과도한 Task 분할과 빈번한 Semaphore 연동은 막대한 코어 파이프라인 낭비와 Context Switch 비용**을 초래합니다.

## 5. Bare-metal Super Loop 시스템의 한계 패러다임

MCU 펌웨어 입문 시 사용하는 `while(1)` 메인 루프 기반 설계 모델은 다음과 같은 이유로 멀티센서 융합의 제약 사항을 만듭니다.

* **Blocking System Call 연쇄 전파:** 루프 내에서 특정 센서(`pulseIn` 등) I/O가 20ms 이상 Blocked 상태에 걸리게 되면 하위 코드 라인에 위치한 관절 모터 제어 PWM 산출 로직 전체가 20ms 연기되는 직렬적 체인 구조를 가집니다.
* **우선순위 역전 및 인터럽트 지연 방치:**
  * **1) 우선순위를 맘대로 매길 수 없어 폴링(Polling) 순서나 Flag 검사 시점에 전적으로 의존하게 됩니다.**
    > **[예시: 폴링 순서 의존의 비극]** 아두이노 `loop()` 안에 `1. 데이터 0.5초 대기 수신 -> 2. 모터 비상 정지` 순서로 코드가 짜여 있다고 가정해 봅시다. 로봇이 지금 당장 넘어지기 직전이라 2번 '모터 정지'를 실행해야 하는데, 1번 라인에서 데이터를 기다리느라 0.5초 동안 묶여(Block) 있다면 모터는 돌지 못하고 로봇은 박살 납니다. 즉, 급한 정도(우선순위)와 무관하게 코드의 '작성 순서'에 목숨을 의존하게 되는 치명적 단점입니다.

    > 폴링(Polling)은 컴퓨터 과학 및 통신 분야에서 하나의 장치(또는 프로그램)가 다른 장치의 상태를 주기적으로 검사하여 일정한 조건이 만족될 때까지 데이터를 송수신하거나 자료처리를 수행하는 방식

  * **2) 중요한 외부 인터럽트(장애물 감지) 발생 시, ISR 내부에서 무거운 연산을 고집하다 전체 인터럽트 반응성(Latency)을 마비시키는 주범이 됩니다.**
    > **[예시: ISR 마비 현상]** 로봇 발이 장애물에 부딪혀 긴급 '충돌 인터럽트(ISR)'가 발생했습니다. 당장 회피 궤적을 짠다고 ISR 함수 내부에서 복잡한 삼각함수 연산(예: 5ms 소요)을 돌려버리면 어떻게 될까요? CPU는 인터럽트(ISR) 처리 중에는 다른 모든 시스템(타이머, 네트워크 수신 인터럽트 등)의 귀를 닫아버립니다. 결국 이 5ms 동안 들어온 조종기 명령 패킷이 싹 다 유실되고 시스템 전역 시간(Global Time)마저 뒤틀려버립니다.
* **해결책 (Deferred Processing):** 반면 RTOS는 커널 큐(Queue/Semaphore)를 통해 **처리 위임 패턴**을 지원합니다. ISR은 단지 "방금 충돌했음!" 이라는 신호(Semaphore)만 단 `1us` 만에 스케줄러에 던져두고 잽싸게 종료(`return`)하여 CPU 전체 마비를 막습니다. 그러면 최상위 우선순위를 배정받은 '긴급 회피 Task'가 이 신호를 받아 별도의 스레드 공간에서 깨어나 안전하게 5ms 수학 연산을 수행하게 됩니다.

    > **[심층 답안: FSM(유한 상태 기계)과의 환상적인 결합]**
    > 네, 유한 상태 기계(Finite State Machine, FSM)로 구현하는 것이 맞고, 사실상 **RTOS + FSM 조합은 로봇 소프트웨어 아키텍처의 절대적인 정석(Standard)**입니다.
    > 
    > ISR(외부 인터럽트) 자체는 어떤 제어 로직도 갖지 않고 단지 `COLLISION_EVENT`라는 깃발(Semaphore) 하나만 꽂습니다. 
    > 잠들어 있던 모터 제어 Task가 이 깃발을 감지하고 깨어나는 순간, 내부의 **상태 머신(FSM) 변수**를 기존 `STATE_WALKING` 상태에서 `STATE_COLLISION_AVOIDANCE` 상태로 즉각적이고 안전하게 전이(State Transition)시킵니다. 
    > 즉, **RTOS는 "언제 인터럽트 마비 없이 깨울 것인가"를 보장하는 하드웨어 시간관리 뼈대**이고, **FSM은 "어떤 행동 절차를 밟을 것인가"를 다루는 소프트웨어 알고리즘의 뇌**로서 분리되어 완벽한 시너지를 이룹니다.

    > **FSM의 한계**: 상태(State)가 5개일 때는 전이(Transition) 화살표가 깔끔하지만, 상태가 20개~50개로 늘어나면 화살표가 거미줄처럼 꼬이는 이른바 '스파게티 상태(State Explosion)' 문제가 발생합니다.

    > **대안 아키텍처**
        >1. Behavior Trees (행동 트리, BT) - 복잡한 로봇의 행동 제어 및 의사결정 위한 계층적 트리 구조 방식입니다. (High-level) 논리를 결정

            BT의 개념: 수행할 행동(TASK)를 (Tree) 구조로 만듭니다. [조건 확인] -> [실패/성공/실행 중] 이 3가지 상태만 반환하는 독립적인 블록(Node)들을 레고 조립하듯이 이어 붙입니다.


        >2. Pub-Sub (발행-구독) / Actor Model (이벤트 구동 설계) - 행동을 실행하기 위해 하위(Low-level) 노드들과 Pub/Sub 또는 Action 통신을 사용하여    명령을 내리고 데이터를 수집합니다 
            
            상태(State)를 없애버리는 철학: "현재 로봇이 무슨 상태인가?"를 중앙에서 관리하는 변수를 아예 없애버립니다.

            동작 방식: 정보(Event)가 발생하면 인터넷 방송국처럼 메시지 큐(Message Queue, ROS 2에서는 Topic)에 데이터를 쏴버립니다(Publish). 그 정보가 필요한 코어/노드들은 알아서 구독(Subscribe)하고 있다가 데이터가 들어오는 순간에만 자기 할 일을 하고 다시 잡니다.

            장점: **ROS 2 아키텍처의 알파이자 오메가입니다.** 카메라 노드, 초음파 노드, 모터 제어 노드가 서로의 '상태'를 전혀 알 필요 없이 완벽하게 독립적으로 개발되고 동작합니다. (추후 우리가 RTOS Stage 1 영역에서 다룰 `Queue`와 `Semaphore`의 전달 로직이 이 거대한 Pub-Sub 시스템의 가장 기초적인 마이크로(Micro) 버전입니다.)

        >3. Continuous Control (연속 피드백 제어 - PID / MPC)
            상태 전이(IF-CASE)의 최소화: 수학적 제어공학 베이스입니다. 만약 장애물과 부딪혔을 때, 소프트웨어적으로 if(충돌) { state = 회피; } 처럼 딱딱 끊어지는 논리 전이를 쓰지 않습니다.
            동작 방식: 장애물과의 거리가 가까워질수록 모터에 역방향 전류(밀어내는 척력 벡터)를 발생시키는 방정식을 설계하여 루프에 넣어버립니다. 로봇은 마치 자석의 같은 극끼리 밀어내듯 수학적 물리 법칙에 따라 스르륵 장애물을 비껴갑니다. (소프트웨어적 상태 머신이 아니라 수식으로 행동을 결정)
